# 7. 비용 산정 기준 (2025년 기준)

비용 산정 기준 및 범위입니다. (2025년 기준, 부가세 별도)

비용은 "화면 수"가 아닌 **기획**·**디자인**·**개발**·**QA**·**보안**·**스토어 배포**·**운영대응 인월**(M/M)로 계산해야 합니다.


## 초기 구축 비용 추정 범위


| 유형 | 비용 | 주요 포함 항목 |
|------|------|---------------|
| **앱 래핑형** | 0.3억 ~ 0.5억 | 기존 웹 재사용, 스토어 등록, 기본 푸시 |
| **하이브리드형** | 0.5억 ~ 0.8억 | WebView + 네이티브 브릿지, 핵심 기능 최적화, 스토어 배포 |
| **크로스플랫폼 표준형** | 1.0억 ~ 1.6억 | 로그인·마이페이지, 주요 서비스·푸시, 간편인증·UI 재설계 |
| **네이티브 확장형** | 2.0억 ~ 3.0억+ | iOS/Android 개별 최적화, 고도 인증·성능, 기기연동 |

## TCO (총 소유 비용) 관점


단순 구축비 외에 **운영 전주기 비용**을 고려해야 합니다.

### 숨은 비용(Hidden Cost)

| 항목 | 설명 | 예상 규모 |
|------|------|----------|
| **콘텐츠 싱크 관리** | 웹과 앱의 콘텐츠/기능 일치 관리 | 운영 인력의 20~30% 추가 공수 |
| **스토어 정책 대응** | 개인정보 처리방침 강화, 심사 기준 변화 대응 | 연 2~4회 재심사/수정 |
| **OS 업데이트 대응** | iOS/Android 메이저 업데이트 시 호환성 점검 | 연 2회, 회당 1~2주 |
| **보안진단** | OWASP MASVS, KISA 모바일 보안 점검 | 회당 500~1,500만 원 |
| **푸시 솔루션** | FCM/APNs 외의 유료 푸시 플랫폼 사용 시 | 월 10~50만 원 |
| **분석 도구** | Firebase, Mixpanel, Amplitude 등 | 월 0~100만 원 (트래픽 기반) |

## 효익의 수치화 (ROI 관점)


"**문의 감소**"를 추상적으로 나열하지 말고, **금액으로 환산**해야 합니다.

| 효익 항목 | 산정 방법 | 예시 |
|-----------|----------|------|
| **콜센터 인건비 절감** | FAQ 앱 내 접근성 향상 → 월 문의 감소 건수 × 건당 처리 단가 | 월 500건 × 2,000원 = 월 100만 원 절감 |
| **SMS/알림톡 비용 절감** | 앱 푸시 대체 → 건당 20~50원 × 월 발송 건수 | 월 10만 건 × 30원 = 월 300만 원 절감 |
| **전환율 상승** | 앱 설치 사용자 대비 구매/신청 전환율 상승 → 건당 평균 객단가 × 증가 건수 | 전환율 2% → 4%, 월 100건 증가 × 5만 원 = 월 500만 원 |
| **재방문율 증가** | 앱 사용자 재방문율 상승 → 광고/마케팅 효율 개선 | 재방문 30% → 60%, CPA 50% 절감 |

> **핵심**: "**앱이 통행료**(운영비)보다 더 많은 가치"를 창출한다는 것을 숫자로 보여주어야 예산 승인이 가능합니다.


## 기능점수(FP) 기반 개발비 산정 절차

기능점수(FP) 전문가를 통한 앱 개발 비용 산출은 **국제 표준(ISO/IEC 20926)** 및 **SW사업 대가산정 가이드**에 따라 진행되며, 6단계의 표준 절차를 거칩니다.

### 6단계 산정 절차

| 단계 | 내용 | 설명 |
|------|------|------|
| **1단계: 사전준비** | 요구사항 정의, 산정 방법 결정 | 개발 대상 앱의 업무 범위와 기능적 요구사항 정의. 정통법(설계 공정 이후) 또는 간이법(기획/발주 단계) 선택 |
| **2단계: 기능점수 산정** | 데이터 기능 + 트랜잭션 기능 측정 | **ILF**(내부논리파일), **EIF**(외부연계파일), **EI**(외부입력), **EO**(외부출력), **EQ**(외부조회) 식별 후 복잡도 가중치 적용 |
| **3단계: 보정전 개발원가** | FP × 기능점수당 단가 | 2025년 기준 기능점수당 단가: **605,784원** |
| **4단계: 보정후 개발원가** | 5가지 보정계수 곱셈 | 규모, 연계복잡성, 성능, 운영환경 호환성, 보안성 반영 |
| **5단계: 직접경비 및 이윤** | 실제 비용 + 이윤(25% 이내) | 출장비, 인쇄비 등 직접경비 + 보정후 개발원가의 25% 이내 이윤 |
| **6단계: 최종 개발비** | 보정후 개발원가 + 직접경비 + 이윤 | 최종 앱 개발 비용 확정 |

## 하이브리드 앱 개발 비용 보정 계수 기준

하이브리드 앱 도입 시 개발 비용 산정을 위한 보정계수는 **SW사업 대가산정 가이드**에 따라 **규모, 연계복잡성, 성능, 운영환경 호환성, 보안성**의 5가지 요소를 기준으로 적용합니다.

### 5대 보정계수 구성 및 기준

| 보정계수 | 기준 | 설명 |
|----------|------|------|
| **규모 보정** | 기능점수(FP) 크기에 따른 생산성 변화 | 500FP 미만 시 1.28 적용 |
| **연계복잡성** | 타 기관과의 연계 수 | 0.88 ~ 1.12 (금융권 연계 많을수록 높아짐) |
| **성능요구 수준** | 응답시간·처리율 등 사용자 요구 복잡도 | 높을수록 계수 상승 |
| **운영환경 호환성** | Android/iOS 등 상이한 운영환경 지원 복잡도 | **하이브리드 앱의 핵심 요소** |
| **보안성 수준** | 암호화·개인정보보호·시큐어코딩 요구사항 수준 | 높을수록 계수 상승 |

### 하이브리드 앱(복수 플랫폼) 개발 시 특이점

| 항목 | 내용 |
|------|------|
| **기능 산정 원칙** | 동일한 기능을 Android와 iOS에서 모두 사용 가능하도록 구현 시, **하나의 소스코드에 동시 구현**하므로 기능점수는 **하나의 기능**으로 산정 |
| **보정계수 조정** | 기능점수는 하나로 산정하되, 두 플랫폼 동시 대응 기술적 난이도를 고려하여 **보정계수를 약간 높게** 설정 |
| **운영환경 호환성 가중** | 이용자의 단말기 환경(OS, 하드웨어)이 이질적인 경우 동등한 서비스 제공을 위한 추가 고려 필요 → 이 항목 보정계수 값 상승 |

### 비용 산출 공식

```
보정후 개발원가 = 보정전 개발원가(FP × 단가) × 규모 × 연계복잡성 × 성능 × 운영환경 호환성 × 보안성
```

> **사례**: 실제 모바일 웹사이트 구축 사례에서 **운영환경 호환성 보정계수 1.06**을 적용하여 이질적 환경에 대한 난이도를 반영했습니다. 하이브리드 앱은 단일 플랫폼 개발보다 관리 복잡도가 높으므로 이러한 보정계수 적용을 통해 실제 투입되는 기술적 노력을 비용에 반영합니다.


## 유지보수 요율 산정 방식 (공공/금융 기준)

하이브리드 앱 전환 시 유지보수 요율 산정 방식은 크게 **시장 관례**에 따른 방식과 **SW 사업 대가산정 가이드**에 따른 정밀 산정 방식으로 나뉩니다. 공공/금융 성격의 프로젝트에서는 주로 가이드에 따른 '**요율제 유지관리비**' 산정 방식을 준용합니다.

### 요율제 유지관리비 산정 절차 (6단계)

| 단계 | 내용 | 설명 |
|------|------|------|
| **1단계** | 유지관리 대상 식별 | 유지보수가 필요한 소프트웨어의 범위 확정 |
| **2단계** | SW 개발비 재산정 (현재가치) | 기능점수(FP) × 현재 기능점수당 단가 (직접경비·부가세 제외) |
| **3단계** | 유지관리 난이도(TMP) 계산 | 시스템 특성에 따라 0~100점 총점수 산정 |
| **4단계** | 유지관리 요율 계산 | 공식: 유지관리 요율(%) = 10 + [5 × (TMP ÷ 100)] → **최소 10% ~ 최대 15%** |
| **5단계** | 직접경비 산정 | 출장비, 인쇄비 등 실제 소요 경비 계산 |
| **6단계** | 최종 유지관리비 산정 | (재산정된 개발비 × 유지관리 요율) + 직접경비 |

### 유지관리 난이도(TMP) 결정 요인

| 항목 | 복잡도 기준 |
|------|------------|
| **유지관리 횟수** | 연간 서비스 요청 횟수 (연 4회 이하 ~ 12회 초과) |
| **시스템 사용자 수** | 일평균 사용자 규모 (내부 50% 초과 또는 대국민 10만 명 초과 시 복잡) |
| **시스템 중요도** | 업무 영향도 및 서비스 중단 시의 파급력 |
| **타 시스템 연계** | 인터페이스를 주고받는 외부 시스템의 개수 |
| **오류복구 신속성** | 장애 발생 시 목표 복구 시간 (6시간 이내 복구 요구 시 복잡) |

### 시장 관례 및 하이브리드 앱 특성

| 구분 | 내용 |
|------|------|
| **일반적인 기준** | 구축비의 **15~20%**를 연간 유지보수 예산으로 책정 (예: 3,000만 원 → 연 450~600만 원) |
| **하이브리드 앱 장점** | 하나의 소스코드로 iOS/Android 동시 관리 → 네이티브 대비 유지보수 비용 상대적 저렴 |
| **상용 솔루션** | 별도 상용 SW(보안 솔루션 등) 포함 시, 서비스 수준에 따라 **12%~20%** 별도 적용 |

> **결론**: 하이브리드 앱 전환 후, 현재 시스템의 기능점수(FP) 기반 재산정 개발비에 **10~15% 요율** 적용이 표준 방식입니다. 다만, 하이브리드 앱은 **OS 업데이트·프레임워크 업데이트**에 민감하므로 이를 고려한 난이도 산정이 필요합니다.
